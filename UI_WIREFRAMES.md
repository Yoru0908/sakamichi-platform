## 〇、全局导航

> **设计原则**：导航采用渐进式设计 (Progressive Disclosure)。
> 初期功能少时保持扁平直链，功能增多后自动演进为分组下拉。
> 路由结构从一开始就支持扩展，但 UI 只展示当前已实现的功能。

### 〇-A. 导航演进策略

#### Phase 1（当前，5 项 + 1 下拉）

顶栏项数 <= 6，全部平铺为直链，仅 工具箱 使用下拉。

```
Logo | 博客 | MSG归档 | 广播 | INS归档 | 工具箱 ▾ | [Moon][Globe][User]
```

#### Phase 2（未来扩展 +データ，6-8 项，开始分组）

新增「数据」板块后，归档类合并为下拉，腾出顶栏空间。

```
Logo | 博客 | 归档 ▾ | 广播 | データ ▾ | ツール ▾ | [Moon][Globe][User]
         │              │               │
         │     ┌────────┴──┐    ┌───────┴───────┐    ┌────────┴────────────┐
         │     │ MSG归档    │    │ Spotify统计   │    │ -- 制作 --          │
         │     │ INS归档    │    │ YouTube统计   │    │ 生写生成            │
         │     └───────────┘    │ ランキング    │    │ MSG生成             │
         │                      │ チャート      │    │ 字幕底图 / 字幕工具 │
         │                      └───────────────┘    │ -- 趣味 --          │
         │                                           │ フォーメーション    │
         │                                           │ サイリウム          │
         │                                           │ 坂道ソート          │
         │                                           └─────────────────────┘
```

**分组触发条件**：当顶栏项超过 6 个时执行合并。具体规则：
- MSG归档 + INS归档 → 合并为「归档」下拉
- 数据类功能 → 独立「データ」下拉
- 工具箱 → 拆分为「制作」和「趣味」两个子分区

#### Phase 3（大规模扩展，8+ 项，Mega Menu）

如果未来功能极多，ツール 演变为多列 Mega Menu：

```
Logo | 博客 | 归档 ▾ | 広播 | 探索 ▾ | [Moon][Globe][User]

                ┌───────────────┴────────────────────────────┐

                │  ──────         ──────────       ────       │
                │  Spotify統計    生写生成          フォーメ    │
                │  YouTube統計    MSG生成           サイリウム  │
                │  Xポスト数      字幕底図          坂道ソート  │
                │  チャート        字幕工具          カレンダー  │
                │  TVer一覧                        My 9 Members│
                └────────────────────────────────────────────┘
```

**交互**：

**路由设计原则**：
- 顶级路由 <= 8 个（`blog`, `messages`, `radio`, `instagram`, `tools`, `data`, `fun`, `auth`）

- 静态页面（about/privacy/terms/contact/links）不在主导航，仅页脚链接

### 〇-G. 导航分组设计决策说明

**为什么不是全部卡片式（区别于 sakamichidb）**：

sakamichidb 是纯工具/数据集合站，适合单页卡片入口。但 Sakamichi Tools 是**社区综合平台**，
包含内容消费（博客/MSG/广播/INS）、创作工具、数据分析等不同心理模型的功能。
用户的使用路径不同：

| 用户类型 | 典型路径 | 心理模型 |
|----------|---------|---------|
| 日常消费者 | 首页 → 博客 → 详情 → 返回 | 浏览/阅读 |
| MSG追踪者 | 首页 → MSG → 推し成员 → 时间线 | 追踪/收藏 |
| 广播听众 | 首页 → 广播 → 実時 → （停留，不跳转） | 沉浸/后台 |
| 创作者 | 首页 → 工具箱 → 生写 → （长时间停留） | 制作/导出 |
| 数据控 | 首页 → データ → Spotify → （切换图表） | 分析/比较 |

不同心理模型需要不同的导航深度和页面结构，因此采用**板块分区 + 顶栏导航**，
而非 sakamichidb 的**单页卡片入口**模式。

**菜单项何时合并的判断标准**：

```
顶栏直链数量 <= 5  →  全部平铺，无需下拉
顶栏直链数量 = 6   →  最不常用的 2 项合并为 1 个下拉
顶栏直链数量 = 7-8 →  按类别分组为 2-3 个下拉
顶栏直链数量 > 8   →  Mega Menu（多列）
```

移动端抽屉不受此限制，始终展示完整的分组列表。

## 一、首页 `/` — 信息流门户

> **设计理念**：首页不是静态功能入口页，而是**动态信息流门户**。
> 用户每次访问都能看到最新内容——最新博客、热门MSG、精选画廊作品、正在放送的广播。
> 这种"新闻站"式首页能有效提升回访率和内容发现率，让用户自然探索各功能板块。
>
> **降级策略**：每个模块独立获取数据，任一模块无数据或接口不可用时自动隐藏，不影响其他模块。

### 一-A. 首页模块结构（上→下排列）

```
模块                | 数据来源               | 降级行为
────────────────────┼────────────────────────┼──────────────────────
1. Hero 轮播        | 博客API + 画廊API      | 回退为静态品牌Banner
2. 正在放送 bar     | 节目表API (FastAPI)     | 显示"下一个节目"或隐藏
3. 最新博客翻译     | 博客API (latest 6)     | 骨架屏 → 超时隐藏
4. 人气MSG          | MSG API (trending 4)   | 未登录显示预览 + 引导注册
5. 画廊精选         | 画廊API (top-liked 4)  | Phase 9+ 前隐藏整个模块
6. 快捷入口栏       | 静态数据               | 始终显示（兜底导航）
```

### 一-H. 动效克制原则（Refined Minimalism）

> 首页有多个信息模块，但**绝不允许多处同时动**。遵循"一个焦点"原则。

| 元素 | 动效 | 克制程度 |
|------|------|---------|
| Hero 轮播 | 5 秒自动切换，crossfade 0.6s | 主自动循环元素 |
| 正在放送 红点 | 缓慢呼吸动画 (pulse 2s) | 仅 `is_live=true` 时亮起 |
| 最新博客卡片 | **无动画**，静态排列 2-4 张 | hover 时微妙上移 + 阴影 |
| 人气MSG卡片 | 横向缓慢自动滚动 (marquee, 8s/轮) | hover/触摸时暂停，显示全部 |
| 画廊缩略图 | 横向缓慢自动滚动 (marquee, 10s/轮) | hover/触摸时暂停 |
| 快捷入口 | **无动画** | hover 时图标变色 |

**MSG/画廊轮换说明**：
- 采用缓慢横向滚动（类似 marquee），不是突变切换
- 速度极慢，用户几乎感觉不到在动，但长时间停留会发现新内容
- hover / 触摸时立即暂停，松开后继续
- 移动端：手动横滑，无自动滚动（节省性能 + 避免触摸冲突）

**规则**：
- Hero 轮播在用户 hover/触摸时**暂停**，不抢夺注意力
- 页面同一时刻**最多 2 处**有缓慢动画（Hero + MSG 或 画廊），但速度差异大不会冲突
- MSG/画廊的 marquee 不与 Hero 同步，错开节奏避免视觉疲劳
- 所有 hover 效果用 `transition: 200ms ease`，不用弹性/回弹动画
- `prefers-reduced-motion: reduce` 时关闭所有自动动画，全部变为静态排列

### 一-I. 首屏性能分层加载

```
首屏渲染 (SSR, 目标 LCP < 1.5s):
  ├── HTML 骨架 + 全局导航 + Hero 静态兜底 Slide    ← 服务端生成
  ├── 最新博客卡片 (2-4 张)                          ← 服务端 fetch, 静态 HTML
  └── 快捷入口栏                                     ← 纯静态

首屏后异步加载 (client:load / client:visible):
  ├── HeroCarousel.tsx    ← client:load, hydrate 后开始轮播
  ├── LiveNowBar.tsx      ← client:load, 首次 fetch + 60s 轮询
  ├── TrendingMSG.tsx     ← client:visible (进入视口才加载)
  └── GalleryPicks        ← client:visible (进入视口才加载)

关键优化:
  - Hero 首张 Slide 由服务端 SSR 直出，JS hydrate 后才启动自动轮播
  - 博客卡片成员头像使用 <img loading="lazy" width="80" height="80">
  - Worker KV 缓存: now-playing 5min, blog/latest 10min, trending 30min
  - 前端不直接查 D1/R2，全部走 Worker API → KV 缓存层
```
